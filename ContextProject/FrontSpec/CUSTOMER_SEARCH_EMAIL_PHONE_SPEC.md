# Спецификация поиска клиента по email и телефону

Дата: 2026-05-11

## Цель
Сделать поиск клиента при создании заказа и в других формах быстрым, понятным и без лишних запросов к backend.

Основной сценарий:
- сотрудник открывает создание заказа;
- начинает искать клиента;
- вводит `email` или `телефон`;
- получает компактный и точный список найденных клиентов;
- выбирает клиента без ручного ввода `Customer ID`.

## Почему текущий подход неудобен
По логам видно, что frontend отправляет слишком много запросов:
- почти на каждое изменение строки;
- даже при невалидном или слишком коротком вводе;
- даже когда введён текст, который не похож ни на email, ни на телефон.

Примеры плохих запросов:
- `phoneNumber=vl`
- `phoneNumber=vlad`
- `phoneNumber=89`
- `phoneNumber=961`

Это создаёт проблемы:
- лишняя нагрузка на backend;
- лишние записи в логах;
- нестабильное поведение autocomplete;
- риск гонок, когда старый ответ приходит позже нового;
- плохой UX для сотрудника.

## Бизнес-требование
Поиск клиента должен работать:
- по `email`;
- по `телефону`.

Поиск по ФИО в этом сценарии не является основным, потому что:
- ФИО может совпадать;
- сотрудник чаще знает телефон или почту;
- email и телефон — более точные идентификаторы.

ФИО можно показывать в результатах как дополнительную информацию, но не использовать как основной триггер поиска.

## Основные UX-принципы
- Не отправлять запросы при любом вводе подряд.
- Искать только тогда, когда ввод похож на реальный `email` или `телефон`.
- Показывать сотруднику понятное состояние поля: что именно нужно ввести.
- Не требовать от сотрудника знания внутренних ID.
- Возвращать короткий список наиболее релевантных клиентов.

## Рекомендуемое поведение frontend

### 1. Определение режима поиска
Frontend должен автоматически определять режим поиска.

#### Поиск по email
Считать, что это email-поиск, если:
- строка содержит символ `@`.

Примеры:
- `ivan@`
- `ivan@mail`
- `ivan@mail.ru`

#### Поиск по телефону
Считать, что это поиск по телефону, если:
- строка не содержит `@`;
- после нормализации содержит только цифры и допустимые символы форматирования;
- итоговая длина цифр достаточна для поиска.

Допустимые вводы:
- `89616521391`
- `+79616521391`
- `8 961 652 13 91`
- `+7 (961) 652-13-91`

#### Что не искать
Не выполнять запрос, если:
- строка короче минимального порога;
- строка не похожа на email;
- строка не похожа на телефон;
- строка состоит из случайных букв без `@`.

Например, не отправлять запросы на:
- `v`
- `vl`
- `vlad`
- `8`
- `89`
- `961`

## 2. Минимальные пороги для поиска

### Для email
Рекомендуется начинать поиск, если:
- строка содержит `@`;
- общая длина строки не меньше `5` символов.

Пример:
- `a@b.c` — уже допустимо
- `ivan@` — допустимо, но можно решать по backend

Практический вариант:
- разрешать поиск сразу после появления `@`, если длина `>= 5`.

### Для телефона
Рекомендуется начинать поиск, если:
- после удаления пробелов, скобок, дефисов и `+` осталось минимум `4` цифры.

Пример:
- `8` → не искать
- `89` → не искать
- `961` → не искать
- `8961` → искать
- `+7961` → искать

## 3. Debounce
Frontend не должен отправлять запрос немедленно на каждый символ.

Рекомендуемое значение:
- `400–500ms`

Рекомендация для проекта:
- использовать `500ms`.

Поведение:
- сотрудник печатает;
- пока печатает без паузы, запрос не уходит;
- как только сделал паузу ~500ms, запускается поиск.

Это сильно уменьшает число запросов и делает интерфейс спокойнее.

## 4. Отмена предыдущих запросов
Если пользователь продолжает печатать до завершения прошлого запроса, предыдущий запрос должен быть отменён.

Зачем это нужно:
- чтобы старые ответы не перетирали новые;
- чтобы не было визуальных скачков результатов;
- чтобы не нагружать backend лишними конкурентными запросами.

Технологически это можно делать через:
- `AbortController`;
- либо через механизм отмены запросов в используемом HTTP-клиенте.

## 5. Кэширование одинаковых запросов
Если пользователь уже вводил тот же нормализованный запрос, не нужно каждый раз заново идти в backend.

Примеры одинаковых запросов:
- `+7 (961) 652-13-91`
- `79616521391`

После нормализации это один и тот же телефон.

Что можно кэшировать:
- ключ запроса;
- результаты поиска;
- короткое время жизни кэша, например `1–5 минут`.

## 6. Нормализация ввода

### Нормализация телефона
Перед отправкой на backend телефон должен приводиться к стабильному виду.

Рекомендуется:
- убрать пробелы;
- убрать скобки;
- убрать дефисы;
- убрать лишние символы;
- сохранить только цифры;
- при необходимости отдельно хранить и отображаемую, и нормализованную форму.

Пример:
- `+7 (961) 652-13-91` → `79616521391`
- `8 961 652 13 91` → `89616521391`

### Нормализация email
Для email достаточно:
- `trim()`;
- приведение к lower-case.

Пример:
- ` Vlad@Mail.Ru ` → `vlad@mail.ru`

## 7. Поведение UI поля поиска
Поле поиска клиента должно вести себя как умный autocomplete.

### До начала поиска
Показывать helper text:
- `Введите email или минимум 4 цифры телефона`

### Во время поиска
Показывать:
- индикатор загрузки;
- текст `Поиск...`

### Если ничего не найдено
Показывать:
- `Клиент не найден`
- кнопку или действие `Создать клиента`

### Если найдено несколько клиентов
Показывать список результатов с такой структурой:
- основной текст: `email` или телефон;
- дополнительный текст: `телефон / email + ФИО`.

Рекомендуемый формат строки:
- `ivan@mail.ru`
- `+7 961 652-13-91 · Иванов Иван`

или:
- `+7 961 652-13-91`
- `ivan@mail.ru · Иванов Иван`

Важно:
- ФИО показывать как подсказку;
- но логика выбора должна опираться на точную сущность клиента.

## 8. Поведение после выбора клиента
После выбора клиента frontend должен:
- сохранить выбранный `customerId` во внутреннем form state;
- показать, какой клиент выбран;
- автоматически подгрузить автомобили клиента;
- сбросить список подсказок в стабильное состояние.

Если выбор очищен:
- `customerId` сбрасывается;
- список автомобилей очищается;
- поле выбора автомобиля блокируется или переходит в пустое состояние.

## Рекомендуемое поведение backend

### Вариант 1 — текущая схема с отдельными параметрами
Например:
- `GET /api/customers/search?email=...`
- `GET /api/customers/search?phoneNumber=...`

Плюсы:
- быстро внедряется;
- не требует больших изменений API.

Минусы:
- frontend вынужден сам определять режим поиска;
- логика частично расползается между frontend и backend.

### Вариант 2 — рекомендуемая схема с единым query
Рекомендуемый endpoint:
- `GET /api/customers/search?query=...`

Тогда backend сам:
- определяет, это email или телефон;
- нормализует ввод;
- ищет нужным способом;
- возвращает результаты.

Плюсы:
- frontend становится проще;
- меньше шансов на ошибку;
- вся поисковая логика живёт в одном месте.

Для проекта это лучший долгосрочный вариант.

## Рекомендуемое поведение backend-логики

### Поиск по email
- нормализовать `query`;
- искать по `lower(email)`;
- поддерживать точное совпадение и, если нужно, частичное начало строки.

### Поиск по телефону
- хранить или вычислять нормализованный телефон;
- искать по нормализованному значению;
- лучше всего поддерживать поиск по началу или по полному совпадению.

### Ограничение результатов
Нужно ограничивать число результатов.

Рекомендуется:
- `limit 10`

Зачем:
- autocomplete не должен возвращать десятки и сотни записей;
- сотруднику удобнее выбирать из короткого списка.

### Сортировка
Рекомендуется сортировать:
- по точности совпадения;
- затем по дате обновления или созданию;
- либо по релевантности, если backend это поддерживает.

## Индексы в БД
Для хорошей производительности без Elasticsearch желательно иметь:
- индекс по `email`;
- индекс по нормализованному телефону.

Если используется PostgreSQL или другая SQL-БД, этого обычно достаточно для текущего сценария.

## Почему не нужен Elasticsearch для этого кейса
Для поиска клиента по `email` и `телефону` Elasticsearch обычно избыточен, потому что:
- это точечный поиск по двум полям;
- SQL с индексами справляется отлично;
- Elasticsearch требует отдельной инфраструктуры и синхронизации.

Elasticsearch стоит рассматривать позже, если появится задача глобального поиска по:
- клиентам;
- заказам;
- автомобилям;
- VIN;
- комментариям;
- файлам;
- свободному тексту.

Для текущего сценария достаточно правильно реализованного backend search endpoint и аккуратного autocomplete на frontend.

## Рекомендуемая реализация для текущего проекта

### На frontend
- debounce `500ms`;
- поиск по email только если есть `@`;
- поиск по телефону только если `digits.length >= 4`;
- не отправлять запросы на обычный текст без `@`;
- отменять предыдущий запрос;
- кэшировать одинаковые нормализованные запросы;
- показывать `Создать клиента`, если ничего не найдено.

### На backend
- лучше перейти на `query`-endpoint;
- либо сохранить текущий API, но жёстко валидировать вход;
- нормализовать email и телефон;
- ограничить выдачу `limit 10`;
- использовать индексы по `email` и `phone`.

## Примеры желаемого поведения

### Сценарий 1 — поиск по email
Ввод:
- `v`
- `vl`
- `vlad`

Результат:
- запросов нет.

Ввод:
- `vlad@ma`

Результат:
- через `500ms` уходит один запрос на email-поиск.

### Сценарий 2 — поиск по телефону
Ввод:
- `8`
- `89`
- `961`

Результат:
- запросов нет.

Ввод:
- `8961`

Результат:
- через `500ms` уходит один запрос на телефон-поиск.

### Сценарий 3 — форматированный телефон
Ввод:
- `+7 (961) 652-13-91`

Результат:
- frontend нормализует ввод;
- backend получает одно корректное значение для поиска;
- пользователю показываются релевантные клиенты.

### Сценарий 4 — клиент не найден
Ввод:
- email или телефон, по которому ничего нет.

Результат:
- показывается `Клиент не найден`;
- доступно действие `Создать клиента`.

## Критерии готовности
Решение можно считать готовым, если:
- frontend не отправляет мусорные запросы на каждый символ;
- поиск по email и телефону работает стабильно;
- сотрудник не вводит `Customer ID` вручную;
- при выборе клиента автоматически подгружаются его автомобили;
- если клиент не найден, есть понятный сценарий создания нового клиента;
- backend не получает бессмысленные запросы вроде `phoneNumber=vl`.

## Следующий практический шаг
Рекомендуемый следующий шаг для проекта:
1. доработать `CustomerLookupField` на frontend;
2. ввести строгие правила валидации и debounce;
3. затем при необходимости упростить API до `query`-формата на backend.
