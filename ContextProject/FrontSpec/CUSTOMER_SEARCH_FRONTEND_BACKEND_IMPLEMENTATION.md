# Реализация поиска клиента для frontend

Дата: 2026-05-11

## Зачем был сделан этот документ
Этот документ описывает, что именно было реализовано на backend для поддержки удобного поиска клиента на frontend при:
- создании автомобиля;
- создании заказа;
- любых формах, где сотруднику нужно быстро выбрать существующего клиента.

Документ написан с акцентом на frontend-разработку:
- какой endpoint теперь использовать;
- какие правила поиска заложены на backend;
- какие запросы frontend должен отправлять;
- какие запросы frontend не должен отправлять;
- как теперь правильно строить autocomplete / lookup field.

---

## Что было не так до изменений
До доработки в проекте уже существовал endpoint поиска клиентов:
- `GET /api/customers/search`

Он принимал параметры:
- `email`
- `phoneNumber`
- `firstName`
- `lastName`

Проблемы старого подхода:

### 1. Логика поиска была частично на frontend
Frontend должен был сам понимать:
- это email или нет;
- это телефон или нет;
- какой именно параметр передавать в backend.

То есть frontend должен был решать, слать ли:
- `?email=...`
- или `?phoneNumber=...`

Это усложняло компонент поиска клиента.

### 2. Backend искал неэффективно
Старая реализация в сервисе делала следующее:
- брала `customerRepository.findAll()`;
- загружала всех клиентов;
- потом фильтровала их в памяти Java.

Это плохо потому что:
- при росте числа клиентов поиск станет дорогим;
- autocomplete не должен перебирать всю таблицу;
- БД умеет искать лучше, чем фильтрация в приложении.

### 3. Не было жёсткой backend-защиты от мусорных запросов
По спецификации frontend не должен отправлять запросы вроде:
- `v`
- `vl`
- `vlad`
- `89`
- `961`

Но backend тоже должен быть устойчивым.
Даже если frontend случайно отправит плохой запрос, backend не должен делать бессмысленный поиск.

---

## Что было реализовано

### 1. Добавлен новый единый endpoint поиска
Теперь backend поддерживает рекомендуемый формат:

- `GET /api/customers/search?query=...`

Это значит, что frontend теперь может использовать один универсальный параметр:
- `query`

И не обязан сам выбирать, какой режим поиска использовать.

### 2. Backend сам определяет режим поиска
Внутри backend теперь есть логика:

#### Если строка содержит `@`
Тогда это считается поиском по email.

Примеры:
- `ivan@`
- `ivan@mail`
- `ivan@mail.ru`

#### Если строка не содержит `@`, но состоит из телефонных символов
Тогда это считается поиском по телефону.

Допустимые символы телефонного ввода:
- цифры;
- пробелы;
- `+`;
- `(` и `)`;
- `-`

Примеры:
- `8961`
- `+7961`
- `8 961 652 13 91`
- `+7 (961) 652-13-91`

#### Если строка обычный текст без `@`
Такой запрос backend игнорирует и возвращает пустой список.

Примеры:
- `vlad`
- `ivan`
- `test`

Это сделано специально, потому что в этом сценарии основной поиск должен быть по:
- email;
- телефону.

---

## Правила, которые теперь заложены на backend

### Поиск по email
Поиск по email выполняется только если:
- строка содержит `@`;
- длина строки не меньше `5` символов.

Пример:
- `a@b.c` -> допустимо;
- `ivan@` -> допустимо;
- `ab@` -> слишком коротко, поиск не выполняется.

Что делает backend:
- обрезает пробелы по краям;
- переводит строку в lowercase;
- ищет по началу email.

То есть запрос:
- `Vlad@Ma`

нормализуется в:
- `vlad@ma`

и ищется как префикс email.

### Поиск по телефону
Поиск по телефону выполняется только если:
- строка не содержит `@`;
- строка состоит только из допустимых телефонных символов;
- после удаления всех нецифровых символов остаётся минимум `4` цифры.

Пример:
- `8` -> поиск не выполняется;
- `89` -> поиск не выполняется;
- `961` -> поиск не выполняется;
- `8961` -> поиск выполняется.

Что делает backend:
- обрезает пробелы по краям;
- извлекает только цифры;
- ищет по началу цифровой последовательности.

Пример:
- `+7 (999)` -> нормализуется в `7999`
- далее backend ищет клиентов, у которых телефон в цифрах начинается с `7999`

## Как именно реализован поиск в базе данных

### По email
Добавлен repository-метод, который делает поиск вида:
- `lower(email) like 'query%'`

Что это даёт:
- поиск не чувствителен к регистру;
- работает autocomplete по началу email;
- не нужно полное совпадение, чтобы показать подсказки.

### По телефону
Добавлен repository-метод, который ищет по цифрам телефона.

В запросе используется нормализация номера прямо на стороне БД:
- из `phone_number` убираются все нецифровые символы;
- затем выполняется поиск по префиксу цифровой строки.

Что это даёт frontend:
- можно вводить телефон в человекочитаемом виде;
- backend корректно ищет даже при форматировании;
- сотруднику не нужно помнить точный формат хранения номера.

---

## Ограничение результатов
Новый backend-поиск ограничивает выдачу:
- максимум `10` записей.

Это важно для frontend autocomplete, потому что:
- не нужно рисовать большие списки;
- проще выбирать клиента;
- меньше лишних данных в сети;
- стабильнее UI.

Если совпадений больше десяти:
- backend всё равно вернёт только первые `10`.

---

## Сортировка результатов
Поиск сортируется так, чтобы более релевантные записи были выше.

### Для email
Сначала выше идут:
- более точные совпадения;
- затем более свежие по `updated_at`;
- затем по `id`.

### Для телефона
Сначала выше идут:
- более точные совпадения по цифровому телефону;
- затем более свежие по `updated_at`;
- затем по `id`.

Для frontend это означает:
- наиболее вероятный нужный клиент обычно будет в верхней части списка;
- autocomplete становится удобнее без дополнительной клиентской сортировки.

---

## Обратная совместимость
Старый endpoint-формат не удалён.

То есть backend всё ещё умеет работать через старые параметры:
- `email`
- `phoneNumber`
- `firstName`
- `lastName`

Но теперь логика контроллера такая:
- если пришёл непустой `query`, используется новый поиск;
- если `query` не пришёл, используется старый режим.

Это сделано для безопасного перехода frontend.

То есть фронт можно переводить постепенно.

## Что это значит для frontend прямо сейчас

## Рекомендуемый способ интеграции
Frontend должен использовать только:
- `GET /api/customers/search?query=...`

И больше не пытаться самостоятельно решать:
- слать `email`;
- слать `phoneNumber`;
- слать что-то ещё.

Backend уже делает это сам.

---

## Рекомендуемое поведение поля поиска клиента

### Что должен делать frontend

#### 1. Держать одно текстовое поле поиска
Например:
- placeholder: `Введите email или минимум 4 цифры телефона`

#### 2. Перед отправкой запроса анализировать ввод
Frontend всё ещё должен отсеивать мусор до запроса.

Рекомендуемая логика:

##### Вариант email
Если в строке есть `@`:
- считать ввод email-поиском;
- разрешать запрос только если длина строки `>= 5`.

##### Вариант телефона
Если `@` нет:
- извлечь цифры;
- разрешать запрос только если цифр `>= 4`;
- дополнительно убедиться, что строка похожа именно на телефон, а не на обычный текст.

##### Иначе
- не отправлять запрос;
- очищать список подсказок;
- показывать helper text.

#### 3. Использовать debounce
Рекомендуемое значение:
- `500ms`

Это значит:
- пользователь печатает;
- запрос отправляется только после паузы;
- backend не получает запрос на каждый символ.

#### 4. Отменять предыдущий запрос
Если пользователь быстро меняет ввод:
- старый запрос надо отменять;
- новый делать только для актуального текста.

Это защищает от гонок, когда:
- старый ответ приходит позже нового;
- UI показывает устаревшие результаты.

#### 5. Кэшировать одинаковые запросы
Если пользователь снова ввёл тот же запрос:
- можно использовать локальный кэш autocomplete;
- не обращаться в backend повторно без необходимости.

## Какие запросы frontend должен отправлять

### Email
Примеры корректных запросов:
- `GET /api/customers/search?query=vlad@ma`
- `GET /api/customers/search?query=ivan@mail.ru`

### Телефон
Примеры корректных запросов:
- `GET /api/customers/search?query=8961`
- `GET /api/customers/search?query=+7961652`
- `GET /api/customers/search?query=+7 (961) 652`

---

## Какие запросы frontend не должен отправлять

### Не отправлять обычный текст
Не надо отправлять:
- `GET /api/customers/search?query=v`
- `GET /api/customers/search?query=vl`
- `GET /api/customers/search?query=vlad`

Причина:
- это не email;
- это не телефонный поиск;
- backend всё равно вернёт пустой список.

### Не отправлять слишком короткий телефон
Не надо отправлять:
- `GET /api/customers/search?query=8`
- `GET /api/customers/search?query=89`
- `GET /api/customers/search?query=961`

Причина:
- цифр меньше 4;
- такие запросы слишком шумные и не дают качественного результата.

### Не отправлять слишком короткий email-запрос
Не надо отправлять запросы, если email-паттерн ещё не достиг минимального полезного порога.

Пример:
- `ab@`

Лучше дождаться длины `>= 5`.

## Ожидаемое поведение UI

### Пока запрос рано отправлять
Показывать подсказку вроде:
- `Введите email или минимум 4 цифры телефона`

### Пока идёт поиск
Показывать:
- `Поиск...`

### Когда есть результаты
Показывать компактный список клиентов.

Рекомендуемая структура одного элемента:
- основная строка: `email` или телефон;
- вторая строка: `Имя Фамилия`;
- при желании третья строка: второй идентификатор, если он полезен.

Пример:
- `vlad@example.com`
- `Владислав Ковригин`
- `+79991234567`

или наоборот:
- `+79991234567`
- `Владислав Ковригин`
- `vlad@example.com`

### Когда ничего не найдено
Показывать:
- `Клиент не найден`
- кнопку или action: `Создать клиента`

---

## Поведение после выбора клиента
После выбора клиента frontend должен:
- сохранить `customerId` в состоянии формы;
- отобразить выбранного клиента как выбранное значение поля;
- загрузить автомобили клиента;
- очистить выпадающий список подсказок.

Если пользователь очистил выбор:
- `customerId` сбрасывается;
- связанные автомобили очищаются;
- выбор автомобиля блокируется или сбрасывается.

## Что важно понимать frontend-разработчику

### 1. Backend уже умеет защищаться от плохих запросов
Даже если frontend ошибётся, backend не начнёт искать по мусорному plain text.

Но это не отменяет frontend-валидацию.
Лучший UX всё равно достигается, если мусорные запросы не отправляются вообще.

### 2. Новый endpoint не означает, что можно слать всё подряд
Да, endpoint стал проще:
- один `query`

Но frontend всё равно должен уважать правила:
- email только при наличии `@`;
- телефон только при `digits >= 4`;
- plain text без `@` не отправлять.

### 3. Старый API пока жив, но для нового UI лучше использовать только `query`
Если новый компонент поиска клиента делается с нуля, лучше сразу строить его только на новом контракте:
- `GET /api/customers/search?query=...`

## Практическая схема интеграции на frontend

### Шаг 1. Пользователь печатает
Поле хранит текущее значение `inputValue`.

### Шаг 2. Frontend решает, можно ли искать
Если:
- есть `@` и длина `>= 5` -> можно искать;
- или цифр телефона `>= 4` -> можно искать;
- иначе искать нельзя.

### Шаг 3. Если искать нельзя
Frontend:
- не шлёт запрос;
- очищает suggestions;
- показывает helper text.

### Шаг 4. Если искать можно
Через debounce `500ms` frontend делает:
- `GET /api/customers/search?query=<value>`

### Шаг 5. При новом вводе
Frontend:
- отменяет прошлый request;
- делает новый только для актуального текста.

### Шаг 6. Показывает результаты
В список попадают до `10` клиентов.

### Шаг 7. После выбора клиента
Frontend:
- сохраняет `customerId`;
- подгружает автомобили клиента;
- закрывает подсказки.

---

## Рекомендуемый итог для frontend
Если кратко, после этих backend-изменений frontend должен строиться так:

- одно поле ввода для поиска клиента;
- один endpoint: `GET /api/customers/search?query=...`;
- debounce `500ms`;
- отмена предыдущего запроса;
- запросы только для валидного email или телефона;
- не отправлять plain text без `@`;
- показывать до `10` результатов;
- при пустом результате предлагать создание клиента.
