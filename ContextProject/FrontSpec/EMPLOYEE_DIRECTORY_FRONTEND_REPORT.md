# Employee Directory: frontend report

Дата: 2026-05-11

## Цель документа
Этот документ описывает, что было добавлено на backend для работы frontend с сотрудниками.

Документ нужен для двух основных frontend-сценариев:
- админ открывает список всех сотрудников;
- админ ищет сотрудника по email через autocomplete / lookup field.

---

## Что было добавлено на backend

Для поддержки frontend была реализована базовая employee directory функциональность.

### 1. Добавлен employee API
Теперь backend поддерживает два endpoint:

- `GET /api/employees`
- `GET /api/employees/search?query=...`

### 2. Доступ ограничен только для ADMIN
Оба endpoint доступны только пользователю с ролью:
- `ADMIN`

Если frontend вызовет эти endpoint от пользователя без роли `ADMIN`, backend должен вернуть ошибку доступа.

Практический вывод для frontend:
- список сотрудников и поиск сотрудников нужно показывать только в админских сценариях;
- для остальных ролей UI либо не должен показывать этот функционал, либо должен быть готов корректно обработать `403 Forbidden`.

### 3. У сотрудника теперь поддерживается email
В локальной employee-модели backend добавлено поле:
- `email`

Это поле используется как основной идентификатор для поиска сотрудника на frontend.

Важно:
- email сейчас добавлен в БД на уровне миграции;
- для старых записей email может отсутствовать, если данные ещё не были заполнены.

Практический вывод для frontend:
- в списке сотрудников и в результатах поиска надо быть готовым к тому, что у части сотрудников `email` может быть `null`, пока миграция данных не завершена.

## Endpoint 1: список всех сотрудников

### URL
- `GET /api/employees`

### Назначение
Используется для:
- админской таблицы сотрудников;
- справочника сотрудников;
- селектов, если нужен полный набор сотрудников.

### Кто имеет доступ
- только `ADMIN`

### Ответ
Backend возвращает массив сотрудников.

Пример структуры одного элемента:

```json
{
  "id": 12,
  "firstName": "Anna",
  "lastName": "Sidorova",
  "email": "anna@example.com",
  "function": "ADMIN",
  "createdAt": "2026-05-11T10:15:30Z"
}
```

### Поля ответа
У сотрудника frontend получает:
- `id` — внутренний идентификатор сотрудника;
- `firstName` — имя;
- `lastName` — фамилия;
- `email` — email сотрудника;
- `function` — роль/тип сотрудника;
- `createdAt` — дата создания записи.

### Сортировка
Список возвращается:
- по `id ASC`

То есть frontend получает сотрудников в стабильном порядке.

### Текущие ограничения
Сейчас endpoint:
- не поддерживает пагинацию;
- не поддерживает фильтрацию по роли;
- не поддерживает сортировку с клиента;
- возвращает весь список сотрудников.

Практический вывод для frontend:
- если сотрудников немного, можно безопасно показывать таблицу или простой список;
- если сотрудников со временем станет много, нужно будет отдельно добавить пагинацию на backend.

## Endpoint 2: поиск сотрудников по email

### URL
- `GET /api/employees/search?query=...`

### Назначение
Используется для:
- autocomplete;
- lookup field;
- выбора сотрудника по email;
- быстрого поиска сотрудника в админских формах.

### Кто имеет доступ
- только `ADMIN`

### Пример корректного запроса
- `GET /api/employees/search?query=anna@ex`
- `GET /api/employees/search?query=manager@shop`

### Формат ответа
Ответ такой же, как у `GET /api/employees`:

```json
[
  {
    "id": 12,
    "firstName": "Anna",
    "lastName": "Sidorova",
    "email": "anna@example.com",
    "function": "ADMIN",
    "createdAt": "2026-05-11T10:15:30Z"
  }
]
```

---

## Как работает поиск

Backend реализует поиск только по `email`.

### Основные правила
Поиск выполняется только если:
- параметр `query` не пустой;
- строка содержит `@`;
- длина строки не меньше `5` символов.

### Если запрос невалидный
Backend возвращает пустой список, если:
- `query` пустой;
- `query` не содержит `@`;
- длина строки меньше `5`;
- строка не похожа на email-поиск.

Примеры запросов, которые НЕ будут искать:
- `a`
- `ann`
- `anna`
- `admin`
- `ab@`

### Если запрос валидный
Backend:
- обрезает пробелы по краям;
- переводит запрос в lowercase;
- ищет по началу email.

То есть поиск работает как prefix search.

Пример:
- email в базе: `anna@example.com`
- запрос: `ann` -> не ищется, потому что нет `@`
- запрос: `anna@e` -> ищется
- запрос: `Anna@Ex` -> нормализуется в `anna@ex` и ищется корректно

---

## Важное ограничение поиска

Сейчас поиск работает только по началу email.

### Что это значит
Если в базе есть email:
- `kov390@gmail.com`

То:
- `kov` — сейчас не ищется, потому что нет `@`
- `kov390@gm` — ищется
- `390@gmail.com` — не ищется
- `gmail.com` — не ищется

Причина:
- backend ищет только prefix search по `lower(email)`;
- поиск по середине строки и по домену сейчас не реализован.

Практический вывод для frontend:
- поле поиска нужно позиционировать как email lookup, а не как свободный полнотекстовый поиск;
- пользователю надо подсказывать, что поиск начинается после ввода email-подобного значения.

---

## Ограничение результатов
Поиск возвращает максимум:
- `10` сотрудников

Это важно для autocomplete, потому что:
- не нужно отрисовывать слишком длинный список;
- ответы компактные;
- UI работает стабильнее.

Если совпадений больше 10:
- frontend получит только первые 10 результатов.

## Рекомендуемое поведение frontend

### Для списка сотрудников
Если нужен полноценный admin employees page:
- вызывать `GET /api/employees`;
- хранить массив сотрудников в таблице или списке;
- отображать колонки:
  - `id`
  - `firstName`
  - `lastName`
  - `email`
  - `function`
  - `createdAt`

### Для autocomplete / lookup
Если нужен быстрый поиск сотрудника:
- использовать `GET /api/employees/search?query=...`
- не использовать `GET /api/employees` для client-side фильтрации всего массива, если цель именно lookup

---

## Как frontend должен валидировать ввод перед поиском
Хотя backend уже умеет возвращать пустой список на невалидные запросы, лучший UX достигается, если frontend не шлёт мусорные запросы вообще.

### Рекомендуемая логика
Отправлять запрос только если:
- строка содержит `@`;
- длина строки `>= 5`.

### Если условие не выполнено
Frontend должен:
- не отправлять запрос;
- очищать suggestions;
- показывать helper text.

Пример helper text:
- `Введите email сотрудника`

или:
- `Начните вводить email, например anna@...`

---

## Рекомендуемая UX-логика autocomplete

### 1. Одно поле ввода
Например:
- placeholder: `Введите email сотрудника`

### 2. Debounce
Рекомендуемый debounce:
- `500ms`

Это защищает backend от запроса на каждый символ.

### 3. Отмена предыдущего запроса
Если пользователь быстро меняет ввод:
- старый request надо отменять;
- новый request делать только для актуальной строки.

### 4. Пустой ввод
Если поле очищено:
- не отправлять запрос;
- очищать dropdown;
- сбрасывать локальный список результатов.

### 5. Пустой результат
Если ничего не найдено:
- показывать `Сотрудник не найден`

---

## Рекомендуемое отображение результата в UI

Для одного элемента autocomplete рекомендуется показывать:
- основной текст: `email`
- дополнительный текст: `Имя Фамилия • function`

Пример:
- `anna@example.com`
- `Anna Sidorova • ADMIN`

Если `email == null`:
- можно скрыть такую запись из lookup-результатов на frontend;
- или показывать fallback вроде `Email не заполнен`.

Но для lookup-поля лучше ориентироваться на сотрудников, у которых email заполнен.

## Рекомендуемый пример frontend-flow

### Сценарий 1: список сотрудников
1. Админ открывает страницу сотрудников.
2. Frontend вызывает:
   - `GET /api/employees`
3. Backend возвращает массив сотрудников.
4. Frontend рендерит таблицу.

### Сценарий 2: поиск сотрудника
1. Админ начинает вводить email.
2. Frontend проверяет:
   - есть ли `@`
   - длина `>= 5`
3. Если условия выполнены, через debounce `500ms` отправляется:
   - `GET /api/employees/search?query=<value>`
4. Backend возвращает до 10 сотрудников.
5. Frontend показывает dropdown.
6. После выбора сотрудника frontend сохраняет объект или `employeeId`, в зависимости от сценария формы.

## Какие ошибки должен учитывать frontend

### 401 Unauthorized
Если токен отсутствует или невалиден.

Что делать:
- стандартный auth flow приложения.

### 403 Forbidden
Если пользователь не `ADMIN`.

Что делать:
- скрывать employee directory заранее, если frontend знает роль;
- либо показывать корректное сообщение об отсутствии доступа.

### 503 Service Unavailable
Если auth/service chain временно недоступен.

Что делать:
- показывать retry/error state.

## Что важно помнить frontend-разработчику

### 1. Это не полнотекстовый поиск
Сейчас employee search — это именно email lookup.

### 2. Не надо слать plain text
Запросы вида:
- `anna`
- `manager`
- `admin`

не имеют смысла и должны отсеиваться на frontend.

### 3. Роль важна
Функциональность сотрудников предназначена для `ADMIN`.

### 4. Email может быть не у всех старых записей
На переходном этапе нужно быть готовым к `null` в `email`.
