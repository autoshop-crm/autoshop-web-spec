## 1. Цель документа

Это ТЗ описывает **минимально рабочую WEB-платформу** для AutoShop CRM, которую нужно реализовать в MVP-объеме без моков и заглушек, с подключением к **реальному backend**.

Главная цель MVP:
- дать сотруднику автосервиса возможность войти в систему;
- просматривать список клиентов;
- открывать карточку клиента;
- работать с автомобилями клиента;
- просматривать и создавать заказы;
- использовать базовые формы и фильтры для ежедневного сценария работы.

Документ фиксирует **scope**, структуру экранов, функциональные требования, API-контракты на уровне интеграции, ограничения и критерии готовности.

## 2. Scope MVP

### 2.1. Что входит в MVP

В MVP обязательно входят следующие блоки:

1. **Авторизация**
   - login по email/password;
   - хранение access token;
   - получение данных текущего пользователя;
   - базовая защита приватных экранов.

2. **Модуль клиентов**
   - список клиентов;
   - поиск/фильтрация клиентов;
   - открытие карточки клиента;
   - просмотр основных данных клиента.

3. **Модуль автомобилей**
   - список автомобилей;
   - просмотр карточки автомобиля;
   - привязка автомобиля к клиенту;
   - базовая форма создания автомобиля;
   - фильтрация автомобилей.

4. **Модуль заказов**
   - список заказов;
   - просмотр карточки заказа;
   - базовая форма создания заказа;
   - отображение статуса заказа;
   - фильтрация заказов.

5. **Базовая инфраструктура frontend-приложения**
   - layout;
   - роутинг;
   - API client;
   - обработка loading/error/empty states;
   - минимальная валидация форм.

### 2.2. Что не входит в MVP

До завершения MVP **не делаем**:
- полноценный клиентский портал для конечного клиента;
- loyalty UI;
- parts UI;
- file storage UI;
- notification UI;
- продвинутую аналитику и dashboard;
- drag-and-drop, realtime, websocket-функциональность;
- сложную RBAC-матрицу на frontend-уровне;
- сложную дизайн-систему;
- офлайн-режим;
- pagination с серверной оптимизацией, если это не нужно для первичного демо;
- создание сложных связей и мастер-сценариев за пределами клиентов/авто/заказов.

## 3. Бизнес-цель MVP

WEB MVP нужен для демонстрации, что система уже покрывает базовый рабочий контур сотрудника автосервиса:

1. сотрудник входит в систему;
2. находит клиента;
3. открывает карточку клиента;
4. смотрит связанные автомобили;
5. создает или открывает заказ;
6. видит текущее состояние заказа.

То есть MVP должен подтверждать, что **backend готов не только на уровне API, но и пригоден для реального пользовательского сценария через web-интерфейс**.

## 4. Пользователи и роли

### 4.1. Основная роль MVP

Основная роль для WEB MVP на этом этапе:
- **Manager / Staff**.

### 4.2. Допустимые роли в первой версии

Если backend уже отдает роли пользователя, web-приложение должно корректно принимать их и отображать в интерфейсе. Но в рамках MVP достаточно поддержать:
- `ADMIN`
- `MANAGER`
- `MECHANIC`

### 4.3. Поведение по ролям

На уровне MVP:
- если пользователь авторизован, он получает доступ к web-интерфейсу;
- если роль существует, она отображается в профиле/шапке;
- если позже потребуется ограничение по ролям, архитектура должна позволять добавить route guards и UI restrictions без переписывания приложения.

Полноценная детальная авторизация на уровне кнопок и секций может быть отложена на следующий этап.

## 5. Формат платформы и общая структура

### 5.1. Тип приложения

Нужно реализовать **SPA web application** для внутреннего использования сотрудниками автосервиса.

### 5.2. Общий layout

Приложение должно содержать:
- **верхнюю панель**;
- **боковую навигацию**;
- **основную рабочую область**.

### 5.3. Верхняя панель

Верхняя панель должна содержать:
- название системы `AutoShop CRM`;
- имя/email текущего пользователя;
- список ролей текущего пользователя;
- кнопку выхода из системы;
- при необходимости — быстрый индикатор состояния API.

### 5.4. Боковая навигация

Минимальный набор разделов:
- `Dashboard` или `Главная`;
- `Клиенты`;
- `Автомобили`;
- `Заказы`.

### 5.5. Основная рабочая область

В основной области должны открываться страницы модулей:
- login page;
- client list;
- client details;
- vehicle list;
- vehicle details/create;
- order list;
- order details/create.

## 6. Технологические требования

### 6.1. Общие требования

Frontend должен:
- быть отделен от backend-логики;
- работать через HTTP API;
- использовать реальные endpoint-ы backend;
- не содержать захардкоженных бизнес-решений, которые противоречат backend.

### 6.2. Обязательные технические блоки

Нужно предусмотреть:
- маршрутизацию между экранами;
- централизованный API layer;
- хранение токена;
- interceptor для Authorization header;
- обработку 401/403/500;
- повторно используемые UI-компоненты форм и таблиц;
- конфигурацию base URL через environment.

### 6.3. Предпочтения по архитектуре

Если стек еще не зафиксирован, рекомендуется:
- `React`;
- `TypeScript`;
- `React Router`;
- `Axios` или аналогичный HTTP client;
- `React Hook Form` или аналог для форм;
- `Zod`/`Yup` или встроенная валидация;
- UI библиотека по необходимости, но без усложнения.

Если стек уже выбран в проекте, необходимо следовать существующему стеку и стилю.

## 7. Модуль Auth

### 7.1. Назначение

Модуль авторизации нужен для входа сотрудника в систему и получения доступа к внутренним разделам web-платформы.

### 7.2. Экран Login

Экран должен содержать:
- поле `email`;
- поле `password`;
- кнопку `Войти`;
- блок ошибки авторизации;
- индикатор загрузки при отправке формы.

### 7.3. Поведение Login

После отправки формы система должна:
1. отправить запрос в AuthService или proxy endpoint;
2. получить access token;
3. сохранить token локально;
4. запросить данные текущего пользователя;
5. перенаправить пользователя на основной раздел приложения.

### 7.4. Logout

При нажатии на кнопку выхода система должна:
- удалить access token из local storage/session storage;
- очистить состояние текущего пользователя;
- перенаправить на страницу login.

Если backend поддерживает logout endpoint, его можно вызывать дополнительно.

### 7.5. Ошибки

Нужно обработать сценарии:
- неверный логин/пароль;
- backend недоступен;
- token невалиден;
- пользователь уже не авторизован.

### 7.6. Definition of Done для Auth

Auth считается готовым, если:
- пользователь может войти в систему через UI;
- token сохраняется и используется в последующих запросах;
- приватные страницы без авторизации недоступны;
- logout корректно завершает сессию.

## 8. Модуль Клиенты

### 8.1. Назначение

Модуль клиентов нужен для быстрого доступа к клиентской базе и перехода в карточку клиента.

### 8.2. Экран списка клиентов

На странице списка клиентов должны быть:
- заголовок страницы;
- строка поиска или блок фильтров;
- таблица/список клиентов;
- действие перехода в карточку клиента;
- индикатор загрузки;
- empty state;
- error state.

### 8.3. Поля списка клиентов

Минимально в списке нужно показывать:
- `id`;
- `firstName`;
- `lastName`;
- `phoneNumber`;
- `email`;
- `createdAt` или дату создания;
- действие `Открыть`.

### 8.4. Фильтрация клиентов

Нужно поддержать базовые фильтры/поиск по полям:
- `email`;
- `phoneNumber`;
- `firstName`;
- `lastName`.

Если backend реализует отдельный search endpoint, UI использует его.

### 8.5. Карточка клиента

Карточка клиента должна отображать:
- ID клиента;
- имя;
- фамилию;
- телефон;
- email;
- дату создания;
- дату обновления;
- список связанных автомобилей или ссылку на раздел автомобилей с фильтром по клиенту.

### 8.6. Связь с другими модулями

Из карточки клиента пользователь должен иметь возможность:
- перейти к автомобилям клиента;
- начать создание автомобиля для этого клиента;
- перейти к заказам клиента, если такой сценарий поддерживается текущим backend.

### 8.7. Definition of Done для клиентов

Модуль клиентов считается готовым, если:
- список клиентов загружается из реального API;
- фильтры работают;
- карточка клиента открывается;
- ключевые данные клиента отображаются без ручных заглушек.

## 9. Модуль Автомобили

### 9.1. Назначение

Модуль автомобилей нужен для просмотра автомобилей клиента и работы с базовой информацией по транспортному средству.

### 9.2. Экран списка автомобилей

На странице должны быть:
- заголовок;
- фильтры;
- список автомобилей;
- кнопка создания автомобиля;
- переход в карточку автомобиля.

### 9.3. Поля списка автомобилей

Минимально нужно отображать:
- `id`;
- `customerId`;
- `brand`/`make`;
- `model`;
- `year`;
- `vin`;
- `licensePlate`;
- действие `Открыть`.

### 9.4. Фильтрация автомобилей

На MVP уровне достаточно базовых фильтров по:
- `customerId`;
- `vin`;
- `licensePlate`;
- `brand`;
- `model`.

Если часть фильтров пока не поддержана backend, UI должен оставить только реально доступные фильтры.

### 9.5. Форма создания автомобиля

Форма создания автомобиля должна содержать минимум:
- `customerId`;
- `brand`/`make`;
- `model`;
- `year`;
- `vin`;
- `licensePlate`.

Требования:
- обязательные поля валидируются на frontend базово;
- ошибки backend отображаются пользователю;
- после успешного создания пользователь видит уведомление и может перейти в карточку автомобиля.

### 9.6. Карточка автомобиля

Карточка автомобиля должна показывать:
- основные данные автомобиля;
- привязку к клиенту;
- возможность перейти к заказам по данному автомобилю, если это поддержано API.

### 9.7. Definition of Done для автомобилей

Модуль автомобилей считается готовым, если:
- список загружается;
- фильтрация в базовом виде работает;
- автомобиль можно создать через UI;
- карточка автомобиля открывается;
- автомобиль можно связать с клиентом через реальные данные.

## 10. Модуль Заказы

### 10.1. Назначение

Модуль заказов нужен для отображения рабочих заказов сервиса и создания новых заказов на основе клиента и автомобиля.

### 10.2. Экран списка заказов

На странице должны быть:
- заголовок;
- фильтры;
- список заказов;
- кнопка создания заказа;
- переход в карточку заказа.

### 10.3. Поля списка заказов

Минимально в списке нужно показывать:
- `id`;
- `customerId`;
- `vehicleId`;
- `status`;
- `createdAt`;
- при наличии — `assignedMechanic` или ответственного;
- действие `Открыть`.

### 10.4. Фильтрация заказов

Базовые фильтры MVP:
- `status`;
- `customerId`;
- `vehicleId`;
- дата создания, если backend это уже поддерживает.

### 10.5. Форма создания заказа

Форма создания заказа должна позволять:
- выбрать или ввести `customerId`;
- выбрать или ввести `vehicleId`;
- указать базовое описание проблемы/работы;
- создать заказ в статусе, допустимом backend по умолчанию.

Если backend еще не поддерживает все поля, UI не должен выдумывать лишние сущности. Форма должна строиться по реально доступному контракту.

### 10.6. Карточка заказа

Карточка заказа должна показывать:
- идентификатор заказа;
- статус заказа;
- ссылку на клиента;
- ссылку на автомобиль;
- дату создания/обновления;
- базовые поля заказа, которые возвращает backend.

### 10.7. Статусы заказа

На MVP уровне достаточно отображать реальные backend-статусы, например:
- `NEW`;
- `IN_PROGRESS`;
- `COMPLETED`.

Если статусов больше, UI должен уметь отображать их без жесткой привязки к ограниченному набору.

### 10.8. Definition of Done для заказов

Модуль заказов считается готовым, если:
- список заказов загружается из API;
- базовые фильтры работают;
- заказ можно создать из UI;
- карточка заказа отображает реальные данные backend;
- связь заказа с клиентом и автомобилем визуально понятна пользователю.

## 11. Навигация и пользовательские сценарии

### 11.1. Обязательные пользовательские сценарии

Система должна поддерживать минимум следующие сценарии:

#### Сценарий A. Вход в систему
1. Пользователь открывает страницу login.
2. Вводит email и password.
3. Получает доступ к приложению.

#### Сценарий B. Поиск клиента
1. Пользователь открывает раздел `Клиенты`.
2. Выполняет поиск по имени, email или телефону.
3. Открывает карточку нужного клиента.

#### Сценарий C. Просмотр автомобилей клиента
1. Пользователь находится в карточке клиента.
2. Переходит к автомобилям клиента.
3. Видит связанный список автомобилей.

#### Сценарий D. Создание автомобиля
1. Пользователь открывает форму создания автомобиля.
2. Заполняет поля.
3. Сохраняет автомобиль.
4. Видит созданную сущность в списке или карточке.

#### Сценарий E. Создание заказа
1. Пользователь открывает раздел заказов или карточку автомобиля.
2. Создает новый заказ.
3. После сохранения видит карточку заказа.

### 11.2. Допустимые упрощения в MVP

Разрешается:
- вводить `customerId` и `vehicleId` вручную, если lookup/select пока не реализован;
- использовать простую таблицу без сложной сортировки;
- использовать простые формы без многошагового мастера;
- использовать базовые toast/alert уведомления.

## 12. Интеграция с backend

### 12.1. Источники данных

WEB MVP должен работать с:
- `AuthService` для авторизации;
- `Core` для клиентов, автомобилей и заказов.

### 12.2. Ожидаемые backend-направления

На уровне интеграции ожидаются следующие группы endpoint-ов:

#### Auth
- `POST /api/auth/login` или `POST /demo-api/auth/login`
- `GET /api/auth/me` или `GET /demo-api/auth/me`
- при наличии: `POST /api/auth/logout`

#### Customers
- `GET /api/customers/{id}`
- `GET /api/customers/search`
- при наличии списка: endpoint для получения списка клиентов или использование search без параметров, если backend это допускает

#### Vehicles
- `POST /api/vehicles`
- `GET /api/vehicles/{id}`
- `GET /api/vehicles/client/{clientId}` или другой endpoint для фильтрации по клиенту
- при наличии — endpoint общего списка/поиска автомобилей

#### Orders
- `POST /api/orders`
- `GET /api/orders/{id}`
- endpoint списка/поиска заказов
- endpoint обновления статуса или редактирования, если он уже реализован

### 12.3. Принцип интеграции

Важно:
- frontend должен адаптироваться к реальному API;
- нельзя выдумывать несуществующие backend-контракты;
- если endpoint еще не готов, это должно быть явно отмечено в backlog/follow-up, а не скрыто моками.

## 13. Требования к состояниям интерфейса

Для каждого экрана списка и карточки обязательны состояния:
- `loading`;
- `empty`;
- `error`;
- `success`.

### 13.1. Loading state
- отображается skeleton или простой loader;
- пользователь понимает, что данные загружаются.

### 13.2. Empty state
- отображается понятное сообщение;
- при необходимости есть CTA-кнопка для создания сущности.

### 13.3. Error state
- отображается понятное сообщение об ошибке;
- есть возможность повторить запрос.

### 13.4. Success state
- данные корректно отрисованы;
- доступны основные действия перехода/создания.

---

## 14. Требования к формам

### 14.1. Общие требования

Все формы MVP должны:
- иметь понятные названия полей;
- показывать обязательность полей;
- валидировать хотя бы базовые ошибки до отправки;
- показывать ошибки backend после ответа;
- блокировать повторную отправку, пока идет запрос.

### 14.2. Минимальная frontend-валидация

#### Login
- обязательный `email`;
- обязательный `password`;
- проверка формата email.

#### Customer filters
- допустим пустой набор полей;
- валидируются только форматы, если поле заполнено.

#### Vehicle create
- обязательные ключевые поля;
- проверка длины/формата VIN и номера, если это нужно.

#### Order create
- обязательный `customerId`;
- обязательный `vehicleId`;
- обязательные поля, реально нужные backend.

## 15. Нефункциональные требования

### 15.1. Производительность

Для MVP достаточно, чтобы:
- страницы открывались без заметных зависаний;
- списки корректно работали на малом и среднем объеме данных;
- приложение не делало лишних повторных запросов на каждый ререндер.

### 15.2. Безопасность

Обязательно:
- не хранить пароли нигде, кроме формы ввода;
- не логировать чувствительные данные в production-стиле;
- передавать access token через Authorization header;
- скрывать приватные страницы от неавторизованных пользователей.

### 15.3. Поддерживаемость

Код приложения должен быть:
- модульным;
- читаемым;
- разделенным по доменам или feature-папкам;
- пригодным для последующего расширения.

### 15.4. Конфигурируемость

Нужно вынести в конфигурацию:
- base URL для Core API;
- base URL для Auth API/proxy;
- режим окружения.

## 20. Критерии приемки

WEB MVP считается принятым, если выполнено все ниже:

1. Пользователь может войти в систему через web-интерфейс.
2. После логина приватные страницы доступны без повторного ввода данных.
3. Раздел `Клиенты` показывает реальные данные backend.
4. Поиск клиента по базовым полям работает.
5. Карточка клиента открывается и отображает ключевые данные.
6. Раздел `Автомобили` позволяет просматривать список и открывать карточку.
7. Автомобиль можно создать через UI, если backend endpoint уже готов.
8. Раздел `Заказы` позволяет просматривать список и открывать карточку.
9. Заказ можно создать через UI, если backend endpoint уже готов.
10. Ошибки API отображаются пользователю в понятном виде.
11. Приложение не использует моки для заявленных MVP-сценариев.
12. Основной demo flow проходит через web UI на реальном backend.
