Главная цель дня:
Довести `AuthService` от состояния "умеет зарегистрировать и залогинить" до состояния "контролирует жизненный цикл токенов и готов к интеграции с Core". К концу дня должно быть понятно, как `Core` будет доверять токенам, как токены отзываются, как проверяются роли и как выглядит стабильный auth-контракт для остальных сервисов.

## Что должно быть готово к концу 11 апреля 2026

- Есть зафиксированный контракт `access token` и `refresh token`.
- `Refresh token` не просто выдается, а управляется предсказуемо:
  - можно обновить пару токенов;
  - старый `refresh token` отзывается;
  - logout инвалидирует текущую сессию.
- Есть минимальная интеграция с `Redis` для blacklist/revoke сценариев.
- Есть endpoint валидации токена для будущей связки с `Core`.
- Роли зафиксированы и seed-данные под них понятны.
- Есть минимум тестов на refresh/logout/validate/invalid token.

## Главный результат дня

Не делать сегодня полноценную enterprise IAM-систему.
Задача дня не в `permissions UI`, восстановлении пароля, rate limiting или OAuth2 Authorization Server, а в том, чтобы сделать надежный и понятный MVP-контур аутентификации для микросервисов проекта.

## Что не делаем в День 3

- Не внедряем сложную permission-модель по операциям.
- Не строим многоуровневый RBAC/ABAC.
- Не делаем полноценный `forgot password`.
- Не делаем внешние логины через Google/Apple.
- Не тратим день на идеальную security-архитектуру, если можно принять временное, но осознанное решение для MVP.

## Архитектурные решения, которые надо зафиксировать утром

Перед началом кодинга надо принять 4 конкретных решения и дальше не дергаться:

1. `Core` на MVP доверяет `JWT`, подписанному общим секретом `AuthService`.
2. `Access token` короткоживущий, `refresh token` длинноживущий.
3. `Refresh token` хранится в БД, а отзыв `access token` живет в `Redis` до истечения срока действия.
4. Для интеграции с `Core` нужен отдельный endpoint проверки токена или хотя бы endpoint, который возвращает claims и роли по валидному `Bearer` токену.

Если эти решения не принять в начале дня, остальная работа начнет расползаться.

### Блок 2. Зафиксировать контракт токенов

Время: `1-1.5 часа`

Что сделать:
- Формально определить состав `access token`:
  - `sub` = `userId`
  - `email`
  - `roles`
  - `type = access`
  - `iat`
  - `exp`
  - `jti` для blacklist-проверок
- Формально определить модель `refresh token`:
  - случайный непрозрачный токен;
  - хранится в БД;
  - связан с пользователем;
  - имеет `expires_at`;
  - имеет `revoked`;
  - при refresh происходит rotation.
- Определить ответ API при логине и refresh:
  - `accessToken`
  - `refreshToken`
  - `tokenType`
  - `accessExpiresIn`
  - `refreshExpiresIn`
  - `userId`
  - `email`
  - `roles`

Что важно решить:
- blacklist храним не по всему JWT в raw-виде, а по `jti` или безопасному идентификатору;
- `refresh token` не используется как `JWT`, если для MVP проще хранить UUID/opaque token;
- `Core` должен уметь полагаться на одинаковый набор claims.

Что должно получиться:
- Есть четкий контракт токена, который не придется передумывать на интеграции с `Core`.

### Блок 3. Redis как хранилище отозванных access token

Время: `1.5-2 часа`

Что сделать:
- Подключить `Redis` к `AuthService`.
- Добавить конфигурацию:
  - host
  - port
  - TTL по сроку жизни токена
- Реализовать сервис blacklist:
  - `blacklistAccessToken(jti, expiresAt)`
  - `isBlacklisted(jti)`
- Выбрать формат хранения:
  - ключ вида `blacklist:access:{jti}`
  - TTL до времени `exp`
- Подготовить fallback-решение:
  - если `Redis` временно недоступен, сервис должен падать предсказуемо, а не молча работать в небезопасном режиме.

Что важно понять:
- Для MVP `Redis` нужен не "потому что модно", а потому что без него logout для уже выданного `access token` фактически не работает.
- Бессмысленно blacklist-ить токен без TTL: это быстро превратится в мусорное хранилище.

Что должно получиться:
- Есть рабочий механизм хранения отозванных `access token` до их естественного истечения.

### Блок 4. Реализовать logout и revoke-сценарии

Время: `1.5-2 часа`

Что сделать:
- Добавить endpoint:
  - `POST /api/auth/logout`
- На logout:
  - извлечь текущий `access token` из `Authorization: Bearer ...`;
  - достать `jti` и `exp`;
  - положить `jti` в `Redis blacklist`;
  - отозвать связанный `refresh token` или текущий переданный `refresh token`;
  - вернуть `200 OK` или `204 No Content`.
- Зафиксировать MVP-правило logout:
  - logout инвалидирует текущую сессию;
  - повторный refresh старым токеном больше не работает;
  - access token после logout больше не считается валидным.

Что дополнительно проверить:
- logout без токена;
- logout с уже отозванным токеном;
- logout с невалидным `Bearer`;
- logout после истечения `access token`.

Что должно получиться:
- Пользователь может завершить сессию предсказуемо, а не только ждать истечения TTL.

### Блок 5. Довести refresh flow до безопасной ротации

Время: `1-1.5 часа`

Что сделать:
- Проверить, что refresh делает именно rotation:
  - старый `refresh token` помечается как `revoked`;
  - создается новый `refresh token`;
  - выдается новый `access token`;
  - старый `refresh token` повторно использовать нельзя.
- Убедиться, что refresh не принимает:
  - неизвестный токен;
  - `revoked` токен;
  - просроченный токен;
  - токен неактивного пользователя.
- Если в логике еще нет явного разделения типа токена, зафиксировать его.

Что должно получиться:
- `Refresh flow` перестает быть "просто еще один endpoint" и становится нормальным жизненным циклом сессии.

### Блок 6. Endpoint валидации токена для Core

Время: `1.5-2 часа`

Что сделать:
- Реализовать один из двух MVP-вариантов:
  - `POST /api/auth/validate`
  - или `GET /api/auth/me`
- Предпочтительный вариант для микросервисной интеграции:
  - `POST /api/auth/validate`
  - на вход получает `Bearer` токен;
  - на выход возвращает:
    - `valid`
    - `userId`
    - `email`
    - `roles`
    - `expiresAt`
    - `tokenType`
- Проверки внутри validate:
  - подпись токена;
  - срок жизни;
  - тип токена = `access`;
  - отсутствие в blacklist;
  - пользователь активен.

Что важно понять:
- Даже если `Core` позже будет валидировать токен локально по shared secret, validate-endpoint все равно полезен как диагностический и резервный инструмент.
- Не надо смешивать validate и refresh в одну логику.

Что должно получиться:
- Есть понятная точка интеграции `AuthService -> Core`.

## Риски дня и как не сорваться

### Риск 1. Уйти в слишком сложную security-архитектуру

Как избежать:
- не строить полноценный OAuth2 server;
- не тратить день на permissions-матрицу;
- держать фокус на token lifecycle.

### Риск 2. Сделать logout "для галочки"

Как избежать:
- logout без blacklist не считать готовым;
- обязательно проверить повторное использование токена после logout.

### Риск 3. Сделать validate endpoint, который проверяет не все

Как избежать:
- validate должен учитывать:
  - подпись;
  - срок действия;
  - blacklist;
  - `type=access`;
  - активность пользователя.

### Риск 4. Забыть про developer workflow

Как избежать:
- заранее продумать, кем входить в систему для smoke и интеграции;
- не оставлять проект без staff-user сценария.
